## Catching up the record — the 06-24 moon tools (undocumented till now)

The last worklog was 06-23, but two tools were built 06-24 and never
logged. Recording them here:

- **`bin/moon-overlay`** — draws `moon-net.json` threads onto any image,
  auto-scaling full-res (capture_full_res) → target resolution so the
  curve transfers onto binned deliverables too (camera is fixed; only
  scale changes). Per-thread colours, points circled, `--extend` for a
  smooth quadratic across the frame.
- **`bin/moon-track`** — auto-tracks the moon per-frame, seeded by ≥3
  hand-marks: fit pixel=f(az), predict per frame, box-search the
  brightest blob, saturation-tolerant centroid. `--append` grows the
  thread with `source="auto"` points. Box-around-prediction makes it
  foreground/ghost-immune.

State of `eclipticam-v3w/moon-net.json` today: **2 hand-marked threads**,
no auto points yet:
- 2026-06-22 dec −9.0, 7 pts, az 160.7–186.1
- 2026-06-24 dec −19.1, 19 pts, az 189.4–213.9

Different declinations AND different az ranges → the curves are
non-parallel and cross — exactly the geometry that will break the
single-arc plate-scale degeneracy once a solver consumes them.

## The moon-net deliverable (today's work)

Extended `moon-overlay` into a publishable, accumulating deliverable and
wired it into `publish-night-cam`.

**Design (decided with Peter):**
- **Backdrop = a FIXED reference-night max-stack**, not tonight's. Reuses
  the existing `reference_nights[0]` in camera.json (2026-06-12 for v3w —
  the same dark/clear night combined-brightness overlays). The net grows
  nightly on a stable image.
- **One fixed S3 key** at the camera prefix (`<cam>/moon-net.png`), like
  `combined-brightness` — not per-night.
- **Runs for both eclipticam cameras**; self-skips cleanly where there's
  no net (v1 today) or no reference stack on disk. All skips exit 0.

**`bin/moon-overlay` changes:**
- `--image` now optional; default backdrop = reference-night max-stack,
  resolved as `<frames_root>/night/<night>/<subcam>_max.fits.fz` (the
  eclipticam `night_layout: percam` puts per-night stacks directly in the
  night dir with a `v3w_` prefix; the `v3w/` subdir there is raw frames).
- Graceful no-net / no-threads / no-backdrop skips.
- **Privacy crop**: the backdrop is real camera pixels, so unlike the
  brightness plot it must be cropped (top `publish_keep_top_px`) and get a
  `.privacy-ok` sidecar, or `publish_night` refuses it. Crop happens AFTER
  drawing — the net points share the backdrop's full-res, top-anchored
  coordinate space, so every kept pixel keeps its y. Checked the v3w
  trails sit at y≈1059–1293 of 2592 — inside the top-82% keep region, so
  nothing is lost. (The low-alt-30 moon sits low on this tilted camera,
  but still above the 232px neighbour-window strip that gets cropped.)
- `--publish` → `s3://astro-berrylands-eu-west-1/<cam>/moon-net.png`.

**`bin/publish-night-cam`:** added a step right after combined-brightness
calling `moon-overlay --camera "$CAMERA" --extend --publish || true`.

**Verified locally on pip** (frames are on this host): rendered v3w — 2
threads, scaled x0.500 onto the binned 2304×1296 reference stack,
privacy-cropped, sidecar written. Eyeballed: both curves land in the
left/lower sky above the rooftop+foliage silhouette, visibly
non-parallel and crossing. v1 skips cleanly (exit 0). Did NOT push to
S3 — that happens on the next dusk publish run (cost negligible).
