# 2026-06-21 — astrocam brightness chart fix, per-camera cloud threshold, v3w night-threshold mismatch

Started from "the astrocam brightness chart looks very odd compared to
eclipticam at the same place." Two sensor-relative values were being
treated as universal constants. Fixed both, then found (while checking
the night-definition change from 06-20) that the v3w capture daemon
ignores camera.json's night threshold entirely — wired it to the config.

## What happened

### astrocam brightness chart was floored at -1 stop

- The combined chart (`brightness-combined.png`) plots
  `log2(mean - pedestal)`. astrocam's deep-night `mean` bottoms out at
  ~512.6 (measured across 6 nights 06-15..20, 6166 deep-night samples:
  global min 512.6, p0.1 512.7, p1 512.8, median 541), but its
  configured `pedestal` was **519** — *above* the dark-frame signal. So
  `mean - pedestal` went negative all night, clamped to the 0.5 log
  floor, and drew a dead-flat line at -1 stop. Only twilight lifted off.
- Root cause: the 519 figure (set 06-15) was sampled too high. A pedestal
  must be <= the darkest mean ever seen. Lowered to **512**.

### The +10 stops clear/cloudy line was meaningless for astrocam

- `SKY_CLEAR_MAX_STOPS = 10.0` was hardcoded in both `bin/nightly-cam`
  (verdict + stack band cap) and `bin/combined-brightness` (chart line),
  shared across all cameras. But "stops above pedestal" is sensor-relative
  (depends on bit depth + pedestal):
  - v3w (16-bit container, pedestal 4380, ~13 stops headroom): +10 sits at
    11% of range — a sensible cloud-glow ceiling; clear troughs are +0..+6.
  - astrocam (raw 10-bit, pedestal 512, only ~10.5 stops to twilight): +10
    is at 69% of range — only reached in twilight, so it could never flag
    a cloudy *dark trough*.
- Made `sky_clear_max_stops` a per-camera camera.json value, read by both
  tools (fallback 10.0 when absent):
  - `astrocam/camera.json`: **4.0** (clear troughs ~0-2, cloudy 06-15 night
    sat ~6-7 — 4 divides them).
  - `eclipticam-v3w/camera.json`: **10.0** explicit (was the global default;
    now documented rather than implicit).
- New astrocam chart reads correctly: clear nights (06-17/18/19/20) dip
  below +4, cloudy nights (06-15 ~6-7, 06-16 ~5) stay above. Published to
  `s3://astro-berrylands-eu-west-1/astrocam/brightness-combined.png`
  (served via fresh presigned URL per page load — no cache purge needed).

Commit `6755a55` (pushed): pedestal 512 + per-camera sky_clear_max_stops.

### v3w night threshold: camera.json was being ignored

- Checked the 06-20 night-definition change (sun -12 -> -10, "night starts
  ~30-40 min earlier"). On the chart, **v3w's curve start is unchanged** —
  first frame is ~23:05 UTC on 06-17 through 06-20 (the 2-min drift is just
  the sun setting later toward the solstice, pure ephemeris).
- Cause: the legacy `eclipticam/capture.py` (still the running capture, per
  06-20 worklog) gates night with its own module constant
  `SUN_ALT_NIGHT_DEG = -14.0` in `decide_mode()`. It never reads
  camera.json's `state.sun_altitude_night_deg`. The -10 edit only affects
  the new `astro-state` daemon, which does NOT gate v3w capture yet. Two
  independent code paths; the config edit had zero effect on capture.
- Also note the threshold is asymmetric: night at -14, day at -10 (intended
  hysteresis), so dusk/dawn arms are cut at different sun depressions.
- **Fix:** `decide_mode()` now takes `night_deg`/`day_deg` params; added
  `night_threshold_deg()` which lazily reads
  `eclipticam-v3w/camera.json` `state.sun_altitude_night_deg` (falls back
  to the module constant). Both v3w and v1 call sites pass it. Single
  source of truth = camera.json. Resolves to -10 now; compiles clean
  (couldn't import-test on pip — `rawpy` is Pi-only — but verified the
  config read and `py_compile`).

### Lens distortion from star trails (eclipticam-v3w): k1 = -0.327

Long interactive session (driven by Peter's eyes on splay — I can't see
the screen). Goal: recover lens distortion from the max-stack alone.

**The physics (Peter's reframe, the key insight):** v3w points south, the
celestial pole is far OFF-frame, so a night's star trail subtends a tiny
angle → the true sky-rotation arc is a near-straight chord. The visible
trail CURVATURE is therefore dominated by LENS DISTORTION, not rotation.
So the right model is "straighten the trails" (fit the radial k1 that
makes them straight), NOT arc-walk/pole-consensus (which assumes rotation
arcs — wrong primitive here). astrocam is the opposite case (pole IN
frame, trails are real rotation circles).

**What made it work (each a correction from Peter):**
- Trails ARE present in the central 1/9th — but ~10x fainter than the
  foreground/edge glow, so a global stretch loses them. Fix: local-
  background subtraction (`bin/separate-trails` local_bg_subtract).
- Star trails vs foreground (rooftops/trees/a parked VAN at the sensor
  centre) separate by elongation + near-horizontal angle, not by an
  array-centre box (the centre points at the van, not sky).
- The real blocker was TRAIL MERGING: plain threshold + connected-
  components fuses parallel trails into one fat blob, so measured "bow"
  was bundle width (a sawtooth artefact), and the 1-param fit RAILED at
  the search bound every time. Fix: a Hessian RIDGE crawler that walks
  each trail's 1-px centreline (`bin/separate-trails`), so parallel
  trails stay separate.

**Result (k1):** with 37 separated single-trail centrelines, the k1 fit gave
a clean interior minimum at **k1 = -0.327** (barrel), objective rising
~1.7x either side — NOT railed. Model: r' = r*(1 + k1*rho^2), centre =
frame centre, rho normalised by half-diagonal, on the binned 1152x648
max stack. **Visually confirmed:** remapping the max image by k1 STRAIGHTENS
the curved trails (Peter: "02_after_k1.fits is excellent" in splay).

**Refined to k1,k2 (the edge needs it):** `edge.png` (top-left corner)
showed residual curvature k1 alone can't remove — the classic r^4 regime.
Re-fit with ~49 trails incl. corner trails to rho~0.95:
**k1 = -0.636, k2 = +0.311** ('barrel with an edge correction' — Peter).
k1 alone over-bends a pure barrel; +k2*rho^4 bends the correction near the
corner to match. Real bowl (objective rises in all directions, soft along
the k1<->k2 trade), not degenerate. Visually 04_k1k2.fits fixes the corner
03_k1only.fits leaves bowed. FREE-CENTRE fit is degenerate on this data
(centre runs to infinity, k1->0) — needs more corner coverage; keep centre
fixed. Fallback if k2 ever looks off: crop to rho<~0.85 and use k1 only.
Both stored in `eclipticam-v3w/camera.json` `distortion` block (now
model=radial_k1_k2).

**Tools (new, committed as WIP):**
- `bin/separate-trails` — Hessian ridge crawler → one centreline polyline
  per trail; `--reg` DS9 sidecar for splay, `--npz` for the fit.
- `bin/fit-distortion-trails` — 1-param k1 bow-minimiser (far-pole case),
  with local-bg, elongation/angle star filter, central-region gating,
  strips/overlay/reg diagnostics. The straighten-model tool. (Its own
  connected-components detection is superseded by separate-trails; next
  step is to feed the separator's centrelines into this fit directly.)

**Caveats / next:** one night, one parameter, centre assumed central.
Per-trail bow is modest (~0.5px) — k1 is well-constrained because 37
trails AGREE, not because any single short trail is dramatically bowed.
Refine by: freeing the centre, adding k2, validating across nights,
wiring separate-trails → fit-distortion-trails into one command, and
applying the same to astrocam (pole-in-frame, the complementary case).
Splay scratch dir: ~/tmp/distortion-view/ (01_before / 02_after_k1 FITS).

### Detrans sweep — new nightly deliverable (v=0.040)

Animate de-translated 10-min windows through the night. Far off-frame
pole → star motion is a near-uniform TRANSLATION over a short window; so
undistort each frame (k1,k2) and shift by -v*(t-t0) along the motion
angle, then max-stack — the 60s streaks register into one sharp high-SNR
streak per window. Sliding 1 min/frame → an animation of the night.

**Velocity (detrans block in eclipticam-v3w/camera.json):** v = 0.040
px/s @ 7.8deg (binned 1152x648). Angle from the separated trail centre-
lines (median 7.8, tight IQR); speed refined VISUALLY from a 10-min
window sweep (0.040 shortest streaks; analytic equator rate 0.047 →
cos(dec)~0.85 → dec~32deg, sane). Peter's ⅓-residual rule (detrans
streak ≈ ⅓ of plain → v within 30% of true) bounded it first.
Independently cross-checked k1/k2 via corner-vs-centre registration
(k1k2 registers the corner best — confirms the distortion fit by a
different mechanism than the bow fit that produced it).

**Tool:** `bin/detrans-sweep` — config-driven (reads distortion + detrans
from camera.json, fallbacks baked in), self-skips cameras with no
`detrans` block. ~0.6s/output-frame after cache warms; full 3h night =
177 frames in ~2 min. Caught + fixed a frame-ordering bug: the night
spans 23→00→01→02 UTC and hour dirs sort lexically 00,01,02,23 — must
sort frames by TIMESTAMP not glob order or windowing breaks.

**Wired into the pipeline:** `publish-night-cam` runs detrans-sweep after
diff-sweep (inside the sweep block, FPS + window/step passed through),
adds `sweep-detrans.mp4` to the publish + privacy-mark lists. Website
(`mywebsite` routes/astro.py + mywebsite.py) displays it as a 4th sweep
video with a caption. Runs on puppy (processing.host) like the other
deliverables. Other cameras (astrocam) self-skip — astrocam is the
pole-IN-frame case, a different (derot) method.

**detrans DEEP (--mode deep) — faint-star depth (Peter's idea):** detrans
gives the MAX SNR path to the darkest stars — register the frames (co-add
faint signal onto the same pixels, √N gain) then subtract the local
background (kill the sky pedestal). Confirmed: skynoise drops ~800x
(10115 max → 12.7 mean-bg); faint stars emerge on a clean dark sky.
Operator = MEAN of registered frames − local-bg (mean averages noise
down, unlike max). Stretch = LINEAR p13→p97.4 (matches splay's
fits_array_to_surface, the percentiles Peter dialed in). Added as
`detrans-sweep --mode deep` → `sweep-detrans-deep.mp4`, a 5th sweep
video, with its own poster. Also added per-video posters (poster-<name>.jpg,
mid-frame of each clip) replacing the shared thumb.

BADLY out of focus on the Module 3 Wide. Lens sweep on the live camera:

### Daytime moon track — v3w pointing cross-check

The moon is visible (just) in v3w DAY frames — a daytime fiducial for the
camera pointing, complementing the night star trails. Tracked it across
hour 19 (2026-06-21, ~20:00-21:00 UTC) by hand-marking in splay.

**Method (the robust version, after false starts):**
1. Predict moon az/alt per frame from ephem (Surbiton, frame EXIF time).
2. Extract a 150x150 crop centred on a rough guess, with the **crop
   ORIGIN encoded in the filename** (`m19_<frame>_x<X0>_y<Y0>.png`) so a
   splay `p`-click converts to full coords unambiguously (full = origin
   + click). This killed a coordinate-mixing bug that had me wrongly
   calling real moon blobs "clouds".
3. Mark the moon, refit pixel = f(az, alt), regenerate crops with the
   cross at the refined prediction; iterate until the cross sits on the
   moon.

**Result:** 7 marked points fit pixel=f(az,alt) **LINEARLY to 0.6px RMS
(max 1.0px)** across the hour — green cross lands on the moon every
frame. A polynomial is NOT warranted (linear is already sub-pixel; 7
points would overfit).

**Key caveat — single-arc degeneracy:** a full pointing+scale fit from one
night's moon arc is degenerate (az/alt correlated along the arc;
boresight and focal length trade off — gnomonic fit gave boresight alt
28 vs 42deg depending on whether k1,k2 applied). So the arc is an
excellent along-arc position predictor but NOT a 2D pointing/scale solve.

**Camera pointing — the clean trick (Peter's insight):** don't fit the
whole geometry. Find the instant the interpolated moon track is at the
CENTRE PIXEL (1152,648); ephemeris az/alt at THAT moment IS the camera
boresight (centre pixel = optical axis, distortion ~0 there). Result:
**boresight az 216.6 +/- 1.9deg, alt 29.7 +/- 1.9deg** — v3w points ~SW,
up ~30deg. The +/-1.9deg is the 44px miss at closest approach (19:58:58
UTC) x 0.044 deg/px; well under Peter's <100px expectation. Degeneracy-
free because it uses one local coincidence, not a global fit.
NB the camera is NOT vertically mounted (it's in a cardboard box) — this
is a FEATURE of the method: the moon measures the ACTUAL physical
pointing incl. the box tilt, no vertical assumption.
To tighten to <1deg: mark the moon in **18/0060.jpg** (19:59:05 UTC, 6s
from the centre-crossing, moon ~44px from centre) — the exact
centre-crossing frame.

**Can the moon test k1,k2? NO** — the moon only spans field-radius
rho 0.03-0.33 (central third); distortion there is 0-29px and slowly
varying, vs 300-430px at the corners (rho 0.85-1.0) where the star
trails constrained k1,k2. Fit changes only 1.64->1.68px with/without
k1,k2 — the moon is blind to edge distortion. **Daytime-moon and
star-trail calibrations are COMPLEMENTARY: moon probes the centre,
stars probe the edge.**

Frames were OUT OF FOCUS (the lens-0.0 bug) so the moon was a soft ~12px
blob — redo on in-focus frames (lens 3.15) for a cleaner track, but the
pointing result already holds.
