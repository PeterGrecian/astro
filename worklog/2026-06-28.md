# 2026-06-28 — fleet shuffle, the 1 GB-Pi lean pipeline, detrans crop, mono dropped

A long multi-day session (06-26 → 06-28). The through-line: **make
eclipticam self-sufficient on its new 1 GB SSD Pi**, sort out where each
camera processes, and harden the deliverables.

## Hardware / processing topology — settled

The SSD migration (eclipticam Pi now has a 110 GB SSD, processes its own
frames locally — see [[project-eclipticam-ssd-migration]]) forced the
question of where everything runs. After a couple of reversals it landed:

- **eclipticam Pi** = captures + processes its own night locally (SSD).
  Self-sufficient. `astro-process` re-enabled, on latest code.
- **astrocam** = captures on its Pi → **writes NFS to muppet** (switched
  2026-06-28: muppet exports `/mnt/bigdisk/astrocam-frames`, the astrocam
  Pi remounts puppy→muppet). muppet processes locally, publishes to S3.
  puppy is out of astrocam.
- **puppy** = **skycam only**. astro-process disabled.
- **muppet** = astrocam. (Briefly held all astro + the web backfill; that
  was interim.)

Detour worth recording: the original split (puppy=skycam, muppet=astro)
**broke astrocam** on 06-26 — astrocam captures to puppy, but muppet
processed from a *frozen* bigdisk copy with no puppy→muppet sync, so the
night never reached it. Reverted to processing astrocam on puppy as a
stopgap (and a deliberate **puppy stability test** — it processed two full
nights, no crash, 66 °C, debunking the "puppy is crash-prone" worry: the
earlier crashes were NOT thermal). Then did the proper astrocam→muppet
capture switch today. See [[project-astro-host-split]].

muppet network: its USB ethernet adapter was swapped; the static profile
was pinned to the dead NIC name → fell to DHCP (.15), breaking its .10 pin
+ NFS role. Reworked to a NIC-agnostic `ethernet-static-10` profile (any
single ethernet → .10).

## The 1 GB-Pi lean pipeline (the big technical work)

The Pi kept OOM-ing / boot-looping processing a night. Root-caused and
fixed, piece by piece — each **measured** to fit 1 GB:

- **ffmpeg/x264** was the encode OOM (`-preset medium` lookahead buffers,
  NOT frame loading — the image2 demuxer already streams). Host-aware:
  Pi gets `veryfast`, `threads=1`, `rc-lookahead=10:sync-lookahead=0:
  bframes=0`. Peak 339 MB. Proven visually equivalent to medium (6×
  amplified diff = black).
- **JOBS / threads host-aware + coupled** — `JOBS=min(cores,RAM_GiB)`,
  `threads=cores/JOBS` so `jobs×threads≈cores`. Fixed both the Pi (8-job
  OOM) and puppy (was 8×7=56 threads → 1×8). 
- **detrans-deep cache leak** — `get_und` evicted by frame-index on-miss,
  so the undistort cache grew unbounded across the night → ~440 MB → swap →
  died as sweep 5 of 5. Fixed: prune to the active window (cache flat at
  ~9, not climbing). Measured: completes on the Pi, swap flat ~140 MB.
  This is the Shake/DEM "bound the working set" principle — the algorithm
  was the problem, not the hardware. SKIP_DETRANS_DEEP now defaults OFF
  (produce it everywhere).

## Deliverables

- **Web-delivery mp4s** wired in: 1280-wide / 60fps / hqdn3d denoise →
  ~1.5 MB vs ~18 MB full-res, visually equivalent, loop+fullscreen on the
  site. Frames are the science, the mp4 is the wow. See
  [[project-astro-web-video]].
- **detrans black-border crop** — undistort (k1=-0.636) leaves a
  lens-shaped black surround. NOT a privacy issue (detrans smears static
  objects). Cropped per the GIMP-tuned framing Peter picked (the "close"
  v2, ~1.84 aspect; both max + deep). 06-27 re-rendered + published.
- **detrans-deep** now a standing deliverable (the max-SNR faint-star
  stacks — the limiting-mag / Neptune-relevant one).
- **mono sweep dropped** — colour + diff + detrans/detrans-deep cover it.
  Stale mono S3 keys auto-delete via a RETIRED list on next publish.
- **skycam blocky video** fixed separately (Berrylands): crf 22→18 +
  aq-strength 1.2. Deployed to puppy.

## Backfill

Web mp4s backfilled to astrocam + eclipticam history. Learned the hard way:
home uplink is ~170 KB/s, and re-uploading the (unchanged) full-res mp4s
made it 9 h — added `--web-only` (transcode existing → web + publish only
the new web files). Also found + fixed: web mp4s were generated but never
in the publish `names[]` list, so they weren't uploaded at all.
