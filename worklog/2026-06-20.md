# 2026-06-20 — eclipticam processing deploy, bug-hunt, data-location strategy, branch merge

Spanned a few days of context (06-18 → 06-20). Set out to find why
`/astro/eclipticam` artifacts stopped appearing; ended up deploying
the whole stage-3 pipeline on the eclipticam Pi, fixing five bugs,
building a data-location resolver, merging unify-cameras to main, and
starting the same deploy on astrocam.

## What happened

### Why eclipticam went stale (06-18)

- Capture was fine — frames landed nightly. But **no stage-3
  processing/publish ran on the eclipticam Pi at all**. The
  4-stage migration removed the old `publish-eclipticam.timer` and
  never deployed its replacement. Last S3 publish was 06-16.
- Pi had **no `.venv` and no ffmpeg** (capture ran on system
  python3). Built the venv (~20 min, fitsio compiles from source on
  the RAM-limited Pi 5; USB storage is I/O-bound, ~20-25 min per
  publish run). Installed ffmpeg.
- Deployed `astro-state.service` + `astro-process.service`
  (+ /etc/default env files), enabled. Backfilled 06-17 manually.
- **SSH cwd trap** cost real time: non-interactive `ssh
  peter@eclipticam.local` lands in `$HOME`, not the repo. Always use
  absolute paths / `git -C /home/peter/astro`.

### Bugs found and fixed

1. `diff-sweep` (commit `1e184bb`): ffmpeg `frame_%05d.jpg` failed on
   sparse window indices (first frame ~00054) → no mp4, yet returned
   0, so publish-night-cam's guarded `cp` died under `set -e` BEFORE
   the S3 upload. Glob the frames + fail loudly.
2. `publish-night-cam` (commit `1ae5d87`): combined-brightness ran
   under system python3 (no matplotlib) instead of the venv.
3. Every-minute re-fire (commit `031303d`): astro-state re-asserted
   pending_process flags in memory every 30s, clobbering
   astro-process's on-disk reset → astro-process re-ran the full
   deliverables every 60s ALL NIGHT (~600×). Fix: astro-state reloads
   on-disk flags each tick. Verified: 0 re-fires / 3 min.
4. Dawn flag never set (commit `031303d`): `_mode_from_sun`
   mislabelled the whole twilight band as "dusk" (altitude alone
   can't tell dusk from dawn) → morning `night→dusk→day` never set
   `dawn_window_complete`. Now uses previous mode.
5. v3w brightness ROI (commit `7ad3425`): the night of 06-19 was
   CLEAR with good stars but gated `verdict=cloudy`, stacked 0/188 —
   a neighbour's ground-level lights low in the frame raised
   whole-frame brightness out of the band. Fix: `brightness_roi:
   {top_fraction: 0.5}` measures gating brightness on the TOP HALF
   only (stack still uses full frame). v3w-only. Re-ran 06-19 →
   `verdict=clear`, stacked 154/188. Real stars recovered.

### Data-location strategy (the big discussion)

- Three independent old/new boundaries: puppy frames (legacy
  `night/<date>/<subcam>` still written; canonical skeleton only),
  S3 (old vs new prefix), storage tiers (USB / S3-cold per
  `whereisallthedata.csv`).
- Decision: **dual-read via a resolver, don't move data for the
  code's sake.** Layout is for humans; software resolves.
- Built `astro/locate.py` + `bin/astro-where` (resolve camera/night
  across roots × layouts), registry-aware (falls back to
  `whereisallthedata.csv` for archived nights), relocation-flexible
  (`cfg.search_roots`). `bin/astro-latest-links` maintains
  `latest-<camera>` symlinks; GC now skips symlinks. Full writeup in
  `design/data-location-and-navigation.md`.
- Deferred the 45 GB legacy puppy tree migration until capture writes
  canonical (else maintaining the split from both sides).
